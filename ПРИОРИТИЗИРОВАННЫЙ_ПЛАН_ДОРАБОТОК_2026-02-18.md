# Приоритизированный план доработок — PROFI-A

Дата: 18.02.2026
Горизонт: 4 спринта (можно сжать до 2-3 при параллельной работе Android + Backend)

## Цель

Закрыть продуктовые разрывы и привести проект к состоянию production-ready по ключевым сценариям:
авторизация/аккаунт, 3D-сканирование, поддержка, полнота контента и локализации.

---

## P0 (критично, блокирует production)

### 1) Auth и аккаунт (backend + Android)

- Реализовать API-контур: login/register/change password/reset password/delete account
- Подключить Android-экраны к API (вместо demo flow)
- Добавить обработку ошибок, состояния загрузки, retry и безопасный logout

**Файлы/модули:** `AuthScreen`, `ChangePasswordScreen`, `ProfileScreen`, `data/remote/*`, `repository/*`  
**Результат:** рабочий жизненный цикл аккаунта без dev-заглушек.

### 2) 3D scan end-to-end

- Реализовать/подключить Python backend по контракту `docs/API_СКАНИРОВАНИЯ_ANDROID_PYTHON.md`
- Закрыть TODO в `RoomScanScreen` (отметка участков)
- Подключить автоподстановку `dimensions` в форму комнаты

**Файлы/модули:** `RoomScanScreen`, `AddRoomScreen`, backend Python API  
**Результат:** сканирование дает измерения, которые попадают в комнату.

---

## P1 (высокий)

### 3) Support backend

- Добавить server endpoint для обращений
- Перевести `SupportScreen` с email intent на API-отправку (вложение + текст)
- Сохранить fallback на email только как запасной вариант

**Результат:** обращения уходят гарантированно, без зависимости от почтового клиента.

### 4) Убрать placeholder-разделы

- Доработать `GeneralDataPlaceholder` и `MaterialsPlaceholder` в `AddWorkTypesScreen`
- Реализовать контент `StagesScreen` вместо `catalog_placeholder`

**Результат:** исчезают "пустые" разделы в пользовательском интерфейсе.

---

## P2 (средний)

### 5) Полная локализация UI

- Довынести hardcoded тексты из Kotlin в `strings.xml`
- Синхронизировать ключи во всех `values*`
- Проверить UX на всех языках (RU/EN/RO/UZ/DE/TG/KK)

**Результат:** единый локализуемый UI без смешения строк в коде.

### 6) Техдолг по отчетам и источнику статуса

- Поддерживать единый "источник правды": `ОТЧЁТ_НЕВЫПОЛНЕННЫХ_ЗАДАЧ.md` + `ОТЧЁТ_ВЫПОЛНЕННЫХ_ЗАДАЧ.md`
- Не вести параллельные дублирующие файлы со статусами

**Результат:** управляемый и прозрачный статус задач.

---

## P3 (низкий)

### 7) UX/доступность и качество

- Унифицировать `contentDescription`
- Проверить контрастность, фокус, talkback
- Расширить тесты на негативные сценарии API и UI

## Критерии готовности

- Нет dev/coming-soon заглушек в ключевых пользовательских сценариях
- Нет блокирующих TODO в runtime-сценариях
- Ключевые функции покрыты smoke-тестами
- Статусы задач и отчеты синхронизированы с фактическим кодом
