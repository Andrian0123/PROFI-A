# Список незавершённых задач по проекту PROFI-A

**Дата составления:** 20.02.2026  
**Источники:** ОТЧЁТ_НЕВЫПОЛНЕННЫХ_ЗАДАЧ.md, STATUS_ЗАДАЧ_CANONICAL.md, документация в docs/, код приложения и ML.

Ниже перечислены задачи, которые на текущий момент **не закончены** или **не реализованы**. Пункты сгруппированы по приоритету.

---

## 1. Критичный приоритет

1. **3D-сканирование (E2E)**  
   Экран отметки участков на плане, визуализация coverage/missing_zones, полная автоподстановка размеров в комнату (в т.ч. площадь пола).  
   Файлы: `RoomScanScreen.kt`, `AddRoomScreen.kt`, `ScanProcessingApi.kt`.

2. **Сброс пароля (Reset password)**  
   Функция сброса пароля по email/ссылке не реализована (backend + экран в Android).

---

## 2. Средний приоритет

3. **Поддержка (Support)**  
   Отправка обращений только через email intent; нет backend endpoint для тикетов.  
   Файл: `SupportScreen.kt`.

4. **Виды работ — контент**  
   Вкладки «Общие данные» и «Материалы» остаются заглушками (`GeneralDataPlaceholder`, `MaterialsPlaceholder`).  
   Файл: `AddWorkTypesScreen.kt`.

5. **Этапы — контент**  
   Экран этапов показывает `catalog_placeholder` («Раздел в разработке»), реальный контент не реализован.  
   Файлы: `StagesScreen.kt`, `StagesReference.kt`.

6. **Локализация UI**  
   Часть текстов по-прежнему захардкожена в Kotlin (ProfileScreen, AddRoomScreen, SupportScreen, AboutScreen и др.); не все строки вынесены в `strings.xml` и синхронизированы по `values*`.

---

## 3. Низкий приоритет

7. **Документация**  
   Рассинхрон между разными отчётами и фактическим состоянием кода; вести единый источник истины (например, STATUS_ЗАДАЧ_CANONICAL.md) и по нему обновлять остальные отчёты.

8. **UX и доступность**  
   Не все элементы имеют и локализованы `contentDescription`; унификация подписей и проверка контрастности/TalkBack не завершены.

9. **Подпись на экране проекта**  
   В отчётах ожидалась подпись о том, что комнаты/сметы/акты формируются индивидуально по проекту; в `ProjectDetailScreen` такого текста нет.

---

## 4. Backend и ML

10. **Сканирование документа (ML)**  
    В модуле `app/ml/document_analyzer.py` используется заглушка `_analyze_image_stub` (размеры A4 по умолчанию, фиксированные метки). Не подключена реальная ИИ-модель для распознавания размеров и содержимого (инженерные коммуникации и т.д.).  
    Файл: `app/ml/document_analyzer.py`, см. `app/ml/README_DOCUMENT.md`.

---

## 5. Улучшения (опционально)

- Подстановка **площади пола (floor_area_m2)** из результата скана в форму комнаты.
- **Визуализация** ответа backend по скану (coverage, missing_zones) на превью.
- **Пункт «Калькулятор» в drawer** — в коде уже есть (`NavigationDrawer.kt`); при расхождении отчётов — зафиксировать в документации актуальное состояние.
- Расширение **тестов**: E2E/интеграционные сценарии «проект → комната → смета → экспорт» и «скан → размеры → комната».
